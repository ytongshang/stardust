# Agent Tool Design

设计 Agent 工具是一门持续迭代的艺术。工具要契合模型的当前能力边界，随模型变强持续调整——不是一次性完成的设计决策。

---

## 核心方法论

**观察输出，不要假设行为。** 调试工具设计问题的唯一方式是真正看模型在使用工具时输出了什么。

工具设计的迭代循环：
```
观察模型输出 → 识别失败模式 → 调整工具边界 → 重新观察
```

---

## 工具设计的关键维度

### 结构约束 vs 自然调用

强迫模型输出特定格式（markdown 结构、特殊标记）不如设计成模型愿意**主动调用**的专用工具。当需要等待用户输入时，工具应能阻塞执行流，而不是依赖 Claude 在输出中嵌入特殊格式。

### 随模型能力升级工具

早期对弱模型有效的约束（如每 5 轮强制提醒任务列表），在模型变强后反而成了限制——模型认为必须严格执行，缺乏灵活性。**工具假设要定期重新审视。**

### Progressive Disclosure（渐进式披露）

避免工具数量膨胀的有效手段：用 [[llm/concepts/agent-skills]] 把能力折叠起来，上下文按需加载，无需增加新工具就能扩展能力。

---

## 设计原则

| 原则 | 说明 |
|------|------|
| 匹配能力边界 | 像设计数学解题工具一样：纸 → 计算器 → 电脑，工具随能力边界变化 |
| 普通触发语气 | 激进指令（`"CRITICAL: You MUST"`）在强模型上导致过度触发，普通描述即可 |
| 阻塞而非格式 | 需要等待的场景用阻塞工具，不要依赖输出格式 |
| 最小专用化 | 一个工具解决一个清晰问题，避免多功能工具导致行为不可预测 |

---

## 常见陷阱

| 陷阱 | 现象 | 解决方向 |
|------|------|---------|
| 约束过强 | 模型僵化执行，缺乏灵活性 | 模型升级后重新评估工具必要性 |
| 约束过弱 | 输出不稳定，结构无法依赖 | 改为专用工具 + 明确触发条件 |
| 依赖输出格式 | 模型随意附加内容破坏格式 | 改用工具调用 |
| 工具数量膨胀 | 上下文被工具描述占满 | 用 Progressive Disclosure 折叠到 Skills |
